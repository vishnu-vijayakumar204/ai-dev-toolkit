Optimisation Tips
Verified across Cursor, Claude.ai, API, and prompting
#01
Use .cursorrules — load context once, not every message
Cursor
Put your stack, conventions, and constraints in .cursorrules. Every chat inherits it free. No extra requests. No pasting. This is the highest-leverage setup step in Cursor — one-time cost, unlimited benefit.
Impact
#02
Stay on Auto mode to avoid burning requests on premium models
Cursor
Auto mode routes to the right model efficiently. Manually picking Opus/o3 for every message burns your 500-request budget fast. Save manual selection for tasks that genuinely need it — complex architecture, hard bugs.
Impact
#03
Only @-mention files directly relevant to the task
Cursor
Every @file goes fully into context. Attaching 10 files when you need 2 wastes context depth — and longer contexts can cost more per request in Max Mode. The model can't ignore what you send.
Impact
#04
Use Notepads for reusable context blocks
Cursor
Store DB schema, API contracts, component patterns as Notepads. Attach via @notepad rather than pasting. Same content, one-time write. Especially valuable when 500 requests is your ceiling — every re-paste in a new chat is wasted overhead.
Impact
#05
New chat per task — don't chain unrelated work
Cursor
Long chained conversations accumulate stale context. Older tokens get compressed or dropped silently — the model may miss constraints from 30 messages back. New task = new chat. Keeps each message clean and effective.
Impact
#06
Use .cursorignore to clean up the codebase index
Cursor
Exclude node_modules, dist, lock files, generated code. Smaller index = faster, more accurate retrieval. Uses .gitignore syntax. A noisy index forces Cursor to read more irrelevant files per @codebase call.
Impact
#07
Composer for multi-file edits, Chat for questions — always
Cursor
Mixing them causes context bleed. A debugging session from earlier in the same chat can confuse the model on a new feature task. Right tool, right job. Composer is stateful for edits; Chat is for exploration and Q&A.
Impact
#08
Use @codebase sparingly — direct @file is cheaper and more precise
Cursor
@codebase triggers a semantic search over the index — not a full dump. Useful when you don't know the files. But when you know the files, direct @file references are faster, more focused, and better for preserving context depth.
Impact
#09
Use Projects to persist context across sessions
Claude.ai
Set a system prompt and attach docs once per Project. Every chat in the project inherits it automatically. Stop re-explaining your stack every new conversation. On a 500-request plan, eliminating repeated setup context is essential.
Impact
#10
Use XML tags to structure complex prompts
Claude.ai
Claude is trained on XML-formatted prompts. Use <context>, <task>, <constraints>. Better structure = more focused output = fewer follow-up messages = fewer requests spent correcting.
Impact
#11
"Complete output, do not truncate" — say it every time
Claude.ai
Truncated output forces a follow-up message. That's an extra request burned. Add "Output the complete code, do not truncate" for any codegen task. You'll save more requests over a month than you'd expect.
Impact
#12
Summarise long threads before pivoting to a new sub-task
Claude.ai
Ask Claude: "Summarise the key decisions from this conversation." Start a new chat with that summary. Keeps context clean, prevents the model hallucinating from contradictions buried deep in a long thread. Cleaner context = fewer correction requests.
Impact
#13
Use Prompt Caching — 90% cost reduction on repeated system prompts
API
Set cache_control: {type: "ephemeral"} on any static system prompt block. Verified 90% cost reduction on cache hits. Directly relevant if you're building anything on top of the Anthropic API alongside Cursor.
Impact
#14
Set max_tokens deliberately — never leave it uncapped
API
For short-answer tasks, set max_tokens low (256). For codegen, set it high (4096+). Uncapped on a high-volume endpoint is how surprise bills happen. Tight max_tokens also prevents verbose padding that doesn't add value.
Impact
#15
Batch API for non-realtime jobs — 50% cost reduction
API
Batches API gives 50% off with 24hr SLA. Perfect for data enrichment, bulk classification, doc processing. Same models, same quality — just queued. If you're processing data in the background, this should be your default.
Impact
#16
temperature=0 for code and structured output tasks
API
Codegen, extraction, JSON output — temperature: 0 makes results consistent and reproducible. Higher temperature wastes tokens exploring variation you don't need for deterministic tasks.
Impact
#17
Front-load constraints — state them before the task
Prompting
"Oh also, it should use TypeScript" at the end of a long prompt wastes the entire previous response. State all constraints before the task. Claude reads the whole prompt before generating — constraints placed first carry maximum weight on output.
Impact
#18
One good example beats three paragraphs of description
Prompting
Show one input→output pair in the exact format you want. Claude pattern-matches immediately. Prose descriptions use 3× more tokens and produce worse output. Show, don't tell — especially on a 500-request budget.
Impact
#19
Negative constraints cut output tokens by 30–50% on code tasks
Prompting
Add "No explanation, just the code" or "No markdown, no preamble". Eliminates verbose wrapping. Fewer output tokens = faster response + less context consumed — especially valuable on fixed request budgets.
Impact
#20
Decompose complex tasks — one focus per prompt
Prompting
"Build a full auth system with OAuth, magic links, and RBAC" in one shot = mediocre output across all parts + likely a follow-up correction request. Sequential focused prompts produce better output AND you can steer between steps before the model goes off-piste.
Impact
#21
Edit the previous response — don't regenerate from scratch
Prompting
If 80% of the output is correct, say: "Keep everything, change only [X]." Re-generating the whole thing burns a full request on work already approved. Targeted edits = faster, cheaper, easier to review. Critical on a 500-request plan.
Impact
#22
Write a CLAUDE.md or AGENTS.md for portable cross-tool context
Prompting
A CLAUDE.md at project root is read automatically by Claude Code at session start. An AGENTS.md works across Cursor, Claude Code, and Gemini CLI. One file, every tool, zero repeated pasting. Your whole team works from the same ground truth.
Impact
#23
Use thinking models only for planning — then hand off to Sonnet
Prompting
Use o3 or Gemini 2.5 Pro thinking mode to produce a plan (write it to a Markdown file). Review it. Then switch to Sonnet for execution. Planning = thinking model (2 req/msg). Execution = normal model (1 req/msg). You get the intelligence without burning double-rate on every message.
Impact
#24
Paste only the relevant code section — not the full file
Prompting
400-line file when the bug is in 30 lines = context waste. Add // rest of file omitted so the model understands the excerpt is intentional. Precise context produces better output and uses less of your context window budget — especially in Max Mode where context depth has a direct cost.
Impact
#25
When stuck, swap models instead of retrying the same one
Prompting
If Claude hesitates, try GPT-4.1. If GPT spins, Claude cuts through. Different models have different failure modes. Switching provider on the same prompt — no other changes — frequently unblocks tasks with no extra prompt engineering needed.

Max Mode — Explained Honestly
The most misunderstood feature in Cursor. What it is, what it costs, and when not to use it.
What Max Mode actually is: Max Mode unlocks the model's full context window. Normal mode limits Cursor sessions to roughly 20,000 tokens for speed and cost. Max Mode expands this — up to 200K for Claude models, up to 1M for Gemini 2.5 Pro. It also enables longer reasoning chains on thinking models. Billing is context-based, not a flat 1 or 2 requests. Large contexts drain your budget unpredictably.
✗ Do NOT use Max Mode for
  • ✗Routine day-to-day coding — cost with no benefit
  • ✗Tasks scoped to 1–3 files — you don't need 200K tokens
  • ✗Every request by default — fastest way to drain budget
  • ✗As a fix for poorly written prompts — fix the prompt first
  • ✗Models that don't expand context in Max Mode — you pay the multiplier for nothing
  • ✗Short debugging sessions on isolated functions
✓ Use Max Mode for
  • ✓Large codebase refactors touching many interconnected files
  • ✓Cross-cutting concerns spanning 5+ files
  • ✓Gemini 2.5 Pro specifically — context jumps to 1M tokens
  • ✓Deep bug hunting where the full call chain matters
  • ✓Critical one-off architecture tasks where quality justifies cost
  • ✓When the task legitimately can't be scoped to fewer files
Verified caveat: Cursor advertises 200K context in Max Mode for Claude models. Community reports consistently show 70K–120K usable in practice due to silent internal truncation for performance. Plan accordingly — Max Mode is not a guaranteed 200K window, and you'll be billed for the extended context regardless.
The Golden Rule
Right context beats more context — always. More context = slower response + higher Max Mode cost + degraded model attention on what actually matters. Instructions from early in a filled context window get progressively less likely to be followed. Ruthlessly include only what's relevant. Precision is a performance optimization.

Same Model, Different Experience — Why?
Claude Sonnet 4.6 in Cursor vs Claude Sonnet 4.6 in Claude Code
Context Management
How context gets assembled
Cursor assembles context through IDE retrieval, file watchers, and its own preprocessing layer. Claude Code sends context directly, cleanly. Same model, different "what the model actually sees." Claude Code's context is leaner and more faithful to what you intend.
Token Overhead
Cursor adds IDE-layer tokens
Cursor injects its own system instructions, file metadata, and retrieval scaffolding into every request. Claude Code's CLI-native approach is significantly leaner. This is why identical tasks use 5.5× more tokens in Cursor — the model isn't smarter in one, the context is just noisier.
Feature Access
Deep Anthropic features vs broad model access
Claude Code has native access to extended thinking, subagents, Agent Teams, 1M context beta, and the full Claude tooling stack. Cursor gives you multi-model flexibility. If you primarily use Claude models, Claude Code extracts more from each model.
Workflow Philosophy
Interactive vs Autonomous
Cursor is built for developers who want to stay in control — see every diff, approve every change. Claude Code is built for developers who want to delegate — describe the outcome, come back to finished work. The model is the same. The trust model is different.

Subagents, Skills & Agent Teams
Claude Code's most powerful — and most underused — features. This is where the real speed multiplier lives.
The core problem they solve: Long sessions accumulate context noise. Every file read, search result, and intermediate reasoning step bloats your main conversation — making subsequent messages slower and more expensive. Subagents and Skills solve this by isolating work into separate contexts that discard when done. Your main session stays clean. The heavy lifting happens in the background.
📄
CLAUDE.md
Always-on context · Session start
A Markdown file at your project root that Claude reads at the start of every session. Whatever you put here, Claude just knows. No prompting required. Best for: coding conventions, commit message format, architecture rules, preferred libraries. If a rule applies to every task — put it here.
Example
Always use TypeScript strict mode.
Never modify files in src/legacy/.
Commit messages: feat: / fix: / chore: prefix.
Stack: Next.js 14, Supabase, Tailwind.
🔧
Skills
On-demand context · Auto-invoked
Reusable instruction sets stored as SKILL.md files in .claude/skills/. Claude scans skill descriptions at startup — when your request matches a skill's description, it loads the full instructions. Think of them as macros. Write a PR review checklist once, invoke it with /review-pr forever. Skills are portable — the same SKILL.md format works across Cursor, Gemini CLI, and Codex CLI too.
Example skill
name: review-pr
description: Review a PR for security, bugs, style
---
[Detailed 20-step review checklist]

Invoke: /review-pr
⚡
Subagents
Isolated workers · Parallel execution
Specialized Claude instances with their own isolated context window, system prompt, and tool permissions. You delegate a task to a subagent — it does the work independently, then returns only the result. The intermediate reasoning, dead ends, and file reads get discarded. Your main session never sees the noise. Can run concurrently — multiple subagents working in parallel.
Example
Main Claude → spawns:
• code-reviewer subagent (read-only tools)
• doc-fetcher subagent (web search only)
• test-writer subagent

All run in parallel. Results return clean.
🤝
Agent Teams
February 2026 · Collaborative agents
Multiple Claude instances that can talk to each other directly and self-assign tasks from a shared list. Unlike subagents (which report to a boss), teammates coordinate mid-task. Best for: one agent writing tests while another refactors the module under test. Anthropic demoed 16 agents building a 100K-line compiler over 2 weeks. Use only when workers genuinely need to coordinate — token usage is significantly higher.
When to use vs subagents
Independent tasks → Subagents
Tasks that need mid-work coordination → Agent Teams

Cost: Agent Teams consume significantly more tokens. Reserve for genuinely parallel complex work.

Skill Security — What to Audit Before Adding Any Skill
Skills run with Claude's full permissions. A malicious SKILL.md can be as dangerous as malicious code in your repo.
Security reminder: Skills are executed with the same trust level as your main Claude session. A SKILL.md file from an untrusted source — or a community repository — can instruct Claude to exfiltrate data, execute destructive commands, or escalate its own permissions. Treat every skill file as code you are committing to production.
💉
Prompt Injection via SKILL.md
High severity · Supply chain vector
A SKILL.md sourced from a public repository can contain hidden instructions in its body — below the legitimate-looking checklist. Claude reads the full file verbatim. Adversarial content appended after a long skill definition is executed with the same trust as your own instructions.

Watch for: HTML comments <!-- ignore previous... -->, instructions disguised as code comments, or content hidden after a page break.
🔐
Sensitive Data Leakage
High severity · Exfiltration risk
Skills with vague or broad tool permissions (e.g. "read all project files") can cause Claude to include secrets, API keys, or private config in intermediate outputs. If a subagent spawned by the skill has web_search or HTTP capabilities, that data can leave your machine.

Audit: What files does the skill read? What network tools does it grant?
⬆️
Privilege Escalation
Medium severity · Permission abuse
A skill can instruct Claude to modify CLAUDE.md, create new SKILL.md files, or change project config. This effectively lets a skill rewrite the rules for your entire session — and persist malicious instructions across future sessions. Skills should not have write access to .claude/ unless explicitly intended.
📦
Supply Chain Risks
Medium severity · Community skills
Community skill repositories have no signing or verification standard as of March 2026. Any skill you import is trusting that author's judgment — and their repo's security. A compromised skill maintainer account can push a malicious update. Pin to a specific commit hash, not a branch tip. Never auto-update skills from external sources.
Common Dangerous Patterns to Avoid
Red flags in any skill definition you're reviewing
Dangerous Pattern #1
Overly broad file access
Instructions like "read all files in the project" without scoping to specific directories. A skill should reference only the files it genuinely needs. Broad reads combined with any outbound capability is exfiltration waiting to happen.
Dangerous Pattern #2
Hardcoded network calls
Skills that instruct Claude to curl or wget to specific external URLs — especially URLs unrelated to the stated purpose. Even a logging endpoint can be an exfiltration channel.
Dangerous Pattern #3
Writing to config files
Any skill that writes to CLAUDE.md, .env, package.json, CI configs, or other skills in .claude/skills/. Persistence of injected instructions across sessions is the goal of this attack class.
Dangerous Pattern #4
Unexplained tool grants
A skill requesting tool permissions beyond what its stated purpose needs. A PR review skill needs only read_file and git_diff. If it also requests web_search or execute_command, ask why.
Dangerous Pattern #5
Long or obfuscated body text
Adversarial content is often hidden deep in a long skill file, past the point where a casual reviewer stops reading. Skills longer than ~100 lines warrant full line-by-line review. Look for HTML comments, unicode lookalike characters, and instructions that contradict the header description.
Dangerous Pattern #6
Spawning sub-skills or sub-agents
A skill that instructs Claude to create additional skill files or spawn subagents with elevated permissions. This is a privilege escalation chain — the original skill may look innocent while the spawned agent does the malicious work in an isolated context you can't directly observe.
Pre-Adoption Security Checklist
Run through this before adding any community or external skill to your project
Security Review — SKILL.md Audit
[ ]
Read the entire file top to bottom. Not just the header. Adversarial instructions are typically placed near the end of a long body.
[ ]
List every tool the skill grants or uses. Map each tool to a stated reason. Reject any tool that has no clear justification in the skill's purpose.
[ ]
Check for any file write instructions. Does the skill ever write to CLAUDE.md, .claude/, .env, or other skill files?
[ ]
Verify source and commit history. Pin to a specific commit SHA. Check whether the repository has had recent unexpected changes.
[ ]
Test in an isolated repo first. Run in a throwaway project with no secrets, no production files, and network monitoring enabled before adding to your main project.
[ ]
Search for HTML comments and unicode tricks. Grep the raw Markdown for <!--, zero-width spaces, and lookalike characters that hide instructions from a visual review.
[ ]
Confirm the skill does not spawn sub-skills or subagents unless you have reviewed the full chain. Dynamic spawning hides the actual execution surface.
[ ]
Apply least-privilege. Strip any permission you don't need. Don't import the full community skill if you only need 30% of it — rewrite it in your own SKILL.md.
The safest approach: Write your own skills from scratch for anything security-sensitive. Use community skills only as a reference — copy the workflow logic, discard the permissions block, rewrite in your own SKILL.md. You understand your own instructions. You can't fully verify someone else's.
